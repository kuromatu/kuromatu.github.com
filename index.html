TDD Boot Camp Tokyo 2013-03に行ってきた #tddbc

Mar 20th, 2013

  • イベントページはこちら
  • http://tddbc.doorkeeper.jp/events/2904

■会場について

  • 会場はいつもの株式会社VOYAGE GROUP
  • 参加人数は40ぐらい。AJITOではぎゅうぎゅう詰めなので、@t_wadaさんの講演とデモだけAJITO、後はパンゲア（一番広い会議室）で実施
  • Wi-fi環境、飲み物、お酒があるのは楽

■@t_wada さんの講演

  • スピリチュアルな話 + 統計で見る有用性の解説。詳しくはスライドで
  • 「6年くらいデバッガを使ってない」「使うときはIE6と戦うときだけ」
  • 仮実装、三角測量、黄金の回転（出典：スティール・ボール・ラン）重要

■TDDデモ

  • TA（Teaching Assistant）の @yattomさん、 @grimroseさんによるペアプロ。お題はFizzBuzz
  • TDDというよりはペアプロのデモという感じ
  • 作業しながらお互いの考えを尊重しどうコミュニケーションしていくか？という意思がよく伝わってきてとてもわかりやすかった

■TDD実践編

  • monmonさんとペア組み。お題はLTSV。LTSVについてはこちら
  • 自分のPCで機能別に交代で実装していく。準備不足で継続的テスト環境構築（vim + rspec + guard）に手間取ったり、vim、tmuxのキーバインドの違いなどであたふたしてた
  • TODO編集 -> テスト実装 -> RED -> 仮実装 -> GREEN -> 本実装 -> GREEN -> git ci -> 交代 というサイクルが安定してからはかなりハイペースで作業できた

■レビュー

  • ペア相互レビューでは自分達と同じくRuby 2.0 + rspecで開発しているチームで実施。お互いにこういう方向で実装しようとしているよ、ってことを共有した。ここでうまく動いてなかったguardの設定を教えてもらって本当に助かった…
  • 全体レビューではRuby, PHP, Java, C#の代表がそれぞれ発表
  • 開発者別・言語別でのアプローチの違いは大きく、自分たちと対比してみるとそれぞれ利点・欠点が見えて面白い。やっぱり個人的にはRubyがいいなーと思ったり

■懇親会

  • 野良LT合戦。個人的には参加者が持ち寄った「こういう場合、どうしたらいいんですかね」というお悩み相談会的なコンテンツがもっとあっても良かったかも、とあるLTを見て思った
  • マイナー言語（TDDBC的な意味で）の発表は「こうがんばったぜ！」という勢いが見れて面白いですね
  • マサカリ（質問的な意味で）が飛んでくる素敵な会場だと再認識

個人的なまとめ

  • TDDを体験として得られたのが最も大きな成果
  • 各種のおすすめ本を読み直したい。テスト駆動開発入門、レガシーコード改善ガイド、リファクタリングRubyエディションなど
  • pry（デバッガ）を使って試す習慣がついているので、もっとテストとして書き残す習慣にシフトさせたい

■他力本願なまとめ

  • @PoohSunnyさんによるまとめ http://matome.naver.jp/odai/2136365798061279801

あとがき

  • 作ったLTSVはこちら
  • 本実装時にmonmonさんに「それRubyでこう書けるんですよ（ドヤ顔）」とよく言っていた気がする

業務系プログラミングへのチャレンジのススメ #vgadvent2012

Dec 14th, 2012

※この記事はVOYAGE GROUP エンジニアブログ : Advent Calendar 2012の14日目の記事として書かれています。

VG Advent Calendarの14日目の担当の @kuromatu です。 最近ではすっかりPHPばかりを書いていますが、今でもpryは手放せないし、pryのようなREPLがPHPにもあればいいのに、と思う自称Rubyistです。

ブログが３日と続かないことに定評がありますが、#vgadvent2012 こそ、このジンクスを打破するチャンスかと思い立ち、急いでOctopressでブログページをでっち上げて書いてみました。

さて表題についてですが、実は昨日サポーターズのイベントで、郵便番号データ（KEN_ALL.CSV）をDBに取り込むというのをライブコーディングとしてチャレンジしてみる、というネタを主に学生に向けて披露していました。

KEN_ALL.CSVは、日本郵便が公開しているken_all.lzh、ken_all.zipを解凍すると出力されるCSVファイルの名前です。全て大文字なところで、多くのプログラマは嫌な予感がしそうな気がしますよね。

郵便番号データダウンロード - 日本郵便

KEN_ALL.CSVと言えば、多くのプログラマにトラウマを植えつけたファイルとして有名で、某所で話題に上がったりしていましたね。 ライブコーディング自体は私の準備不足と、お題としてそもそもどうよというところで色々失敗してしまいましたが、それはまた別のお話。後日話します。

この「KEN_ALL.CSVを取り込む」という問題を解いていてふと思ったのが、この手の問題は定番の手法が思ったより公開されていない、というところでした。 最近になってちらほらと公開されてきたようですが、まだまだ公開され成熟した方法は少ないように思えます。

KEN_ALL.CSVに関しては、PerlやRuby on Railsでは公開されていますね。私自身もとても参考になりました。この場を借りて御礼申し上げます。

yappo/p5-Parse-JapanesePostalCode · GitHub

kazuhisa/ken_all · GitHub

というわけで何が言いたいかというと、冬休みの宿題として、アルゴリズム的なプログラミングへのチャレンジもいいけど、KEN_ALL.CSV的な業務的なプログラミングへのチャレンジもアリなんでない？というところです。

私もWebエンジニアになってから、様々なCSVやXMLなどのデータを社内外のユーザに向けて利用しやすいような形式に変換・加工しWebアプリケーションの一部に組み込む、という仕事をよくやってきました。 私が株式会社VOYAGE GROUPに入ってからも何度かありました。

決して楽しい仕事ではないかもしれませんが、とてもよく発生しうる仕事ですので、ある程度慣れておいて損は無いと思っています。

KEN_ALL.CSV以外のお題としては、総務省や国土交通省など、国が提供しているデータはいかがでしょうか。もちろん、利用規約をよくお読みになった上での話ではありますが。

総務省｜電子自治体｜全国地方公共団体コード

国土数値情報　医療機関データの詳細

これらをうまいこと扱えると、仕事やマッシュアップイベントで活躍したり、さらに共有できれば社会貢献度が高まって良いんじゃないかと思います。

休みの日まで業務意識でプログラミングしたくねーよ！というもっともなご意見はあるかと思いますが、自分で取り組んでみたら案外面白かったので、皆様も冬休みにチャレンジしてみてはいかがでしょうか。

私も宿題として、この冬中にKEN_ALL.CSVを解決して自分のGithubリポジトリに公開します。 もし上がってなかったら、石やらマサカリやらを投げておいてください…。

明日の担当は@brtriverさんです。
